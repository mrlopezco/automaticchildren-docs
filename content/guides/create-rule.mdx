# Create a rule

This guide walks you through creating a rule in **Project Settings > Automatic Children**: naming it, setting conditions, and defining child work items with field mappings and placeholders.

## 1. Open the rule list

1. Go to **Project Settings** (gear icon in your project).
2. Open **Automatic Children**.
3. On the **General** tab, you’ll see the list of **Rules** and **Cascades**. Click **Add rule** (or the equivalent button to create a new rule).

<Callout type="info" emoji="🖼️">
  **Screenshot placeholder:** General tab showing list of rules and cascades with Add rule button.
</Callout>

## 2. Name and enable the rule

- Enter a **name** that describes when and what the rule creates (e.g. “Tasks for User Story”, “Bug analysis task”).
- Ensure the rule is **enabled** if you want it to run when you use **Automatic Child Items** from a work item. You can disable it later without deleting it.

## 3. Set conditions (when the rule runs)

- **Work item types:** Select one or more types (e.g. User Story, Bug). The rule runs only when the parent work item is one of these types.
- **States (optional):** Select one or more states (e.g. New, Active). If set, the parent must be in one of these states.
- **Field condition groups (optional):** Add groups of field conditions (e.g. “Area Path equals MyTeam”). Use the field picker to choose the field, operator, and value. Conditions inside a group are ANDed; at least one group must match (groups are ORed). See [Condition operators](/reference/conditions) for operator list.

<Callout type="info" emoji="🖼️">
  **Screenshot placeholder:** Rule editor — conditions section with work item types, states, and a condition group (field, operator, value).
</Callout>

## 4. Define child work items

- Add one or more **child work item** definitions.
- For each definition:
  - **Child work item type:** Choose the type to create (e.g. Task, Bug).
  - **Field mappings:** Map fields to values. Use placeholders like `{{System.Title}}` or `{{System.AssignedTo}}` to copy from the parent. At least **Title** (or your process’s required title field) should be set. Use the field picker to choose fields and insert placeholders if the UI supports it.

Example: Child type **Task**, Title: `Task for {{System.Title}}`, Assigned To: `{{System.AssignedTo}}`.

<Callout type="info" emoji="🖼️">
  **Screenshot placeholder:** Rule editor — child item definitions with type selector and field mappings (e.g. Title with placeholder).
</Callout>

## 5. Save

- Click **Save** (or equivalent). The rule appears in the list and will be evaluated when you run **Automatic Child Items** from a matching work item.

## Tips

- Start with one work item type and one child type; add more conditions and children once it works.
- Use [Placeholders](/concepts/placeholders) to keep child titles and key fields in sync with the parent.
- If the rule doesn’t match, check work item type, state, and field conditions against the work item you’re opening.
